# v2.0 技术路线图

> 从"小模型长文本QA"到"老设备本地跑V4-Pro级超大模型"

## 目标模型

**DeepSeek V4-Pro**
- 总参数：1.6T（MoE，384 路由专家 + 1 共享专家）
- 激活参数：49B/token（每 token 仅激活 6 个专家）
- 稀疏度：97%（每 token 只用 3% 参数）
- 层数：61
- 权重大小：865GB（HF 格式）

## 三条破局路线

### 路线 A：预判路由（Expert Predictive Routing）

**问题**：V4-Pro 有 384 个专家，865GB 权重，但每次推理只用 6 个。如果提前知道用哪些，只需加载 20-50GB。

**方案**：zhongxing 在压缩文本的同时分析问题领域和文本主题，输出"专家预判表"。

**zhongxing 独创**：现有预判路由都是模型内部的（推理时预测下一步专家），zhongxing 是在推理前从文本语义层面预判——"压缩端预判"vs"推理端预判"的本质差异。

**预期收益**：865GB → 20-50GB 权重加载量，IO 减少 94-97%

### 路线 B：深度可变推理（Variable-Depth Inference）

**问题**：61 层全跑，每层都要从 SSD 搬专家权重，简单问题也被迫做全量 IO。

**方案**：zhongxing 给问题打难度标签，简单只跑 15-30 层，复杂跑满 61 层。

**预期收益**：简单问题 IO 量砍 50-70%，复杂问题保持全量

### 路线 C：蒸馏子网络（Distilled Sub-Network）

**问题**：即便只加载 20-50GB 专家权重，SSD 换页仍有延迟。

**方案**：用 V4-Pro API 跑领域 QA 对（10K+ 条），微调本地 14B，使其在该领域逼近 V4-Pro 输出。

**预期收益**：高频领域无需加载 V4-Pro 专家，本地 14B 直接回答，延迟 < 1 秒

## 终极架构：三线合流

```
用户提问
    ↓
zhongxing 2.0 预处理引擎
    ① 文本压缩：1M原文 → 5-10K机语（20x压缩）
    ② 领域识别：输出专家预判表（8-20/384个专家）
    ③ 难度标签：简单/中等/复杂 → 决定推理深度
    ④ 缓存决策：是否命中已蒸馏子网络？
    ↓
┌────────────┴────────────┐
命中蒸馏模型          未命中，走 V4-Pro
本地14B直接回答      Flash-MoE 式 SSD streaming
延迟<1s              只加载预判的 8-20 个专家
精度≈V4-Pro          按难度标签跑 15-61 层
                     ↓
              蒸馏14B做 draft model
              V4-Pro 只验证 draft
              → 2-3× 加速
    └────────────┬────────────┘
                ↓
             结果返回
```

## 降级策略

预判路由可能失败，必须有优雅降级机制：

1. Level 1：只加载预判的专家集 S（理想路径）
2. Level 2：如果 S 覆盖不足 → 从缓存补加载
3. Level 3：如果缓存未命中 → 全量 SSD 换页（慢但不崩）

**预判准确率门槛**：精确率 > 60%（省钱）且 召回率 > 80%（不频繁补加载）
